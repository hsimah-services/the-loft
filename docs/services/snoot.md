# `snoot`

> Beszel agent — one container per host with `network_mode: host`, reporting system + Docker metrics to the [houstn](houstn.md) hub.

## Overview

`snoot` (a dog's nose — sniffs each host) is the Beszel agent. It runs on every fleet host with `network_mode: host` so it can see the kernel's full picture of CPU, RAM, disk, and network plus the Docker socket for per-container stats. The hub side — Beszel itself — lives in [houstn](houstn.md) on [space-needle](../hosts/space-needle.md). One agent, one hub, all four hosts.

Snoot lives in its own service rather than as a `houstn` profile because it runs on **every** host, including hosts that don't run the houstn `metrics` profile (currently a moot distinction since metrics also runs everywhere — but the agent has different network requirements: host networking rather than bridge).

## Architecture

Single container per host, `henrygd/beszel-agent`, host network, listens on TCP 45876:

| Property | Value | Reason |
|----------|-------|--------|
| `network_mode: host` | yes | Agent reports the host's actual interfaces / stack |
| Port | `PORT=45876` | Hub connects to `<host>:45876` |
| Docker socket mount | `/var/run/docker.sock:/var/run/docker.sock:ro` | Per-container CPU/RAM/network stats |

The agent is stateless — no `/opt/snoot` directory. All state (which systems exist, history, dashboards) lives on the hub at `/opt/houstn/beszel/data`.

### How the hub reaches each agent

The hub container (`beszel`, on the `loft-proxy` bridge) configures one **system** per host through the Beszel UI. Each system has a host:port:

| Host | Hub's connection target | Why |
|------|-------------------------|-----|
| space-needle | `host.docker.internal:45876` | The hub and the local agent live on the same machine. The hub is on a bridge; from inside its container `localhost` is the container, not the host. `host.docker.internal:host-gateway` (in the houstn compose's `extra_hosts`) resolves to the host gateway from inside the bridge. |
| viking | `viking:45876` | Resolved via [mushr](mushr.md)-dns A record |
| fjord | `fjord:45876` | Same |
| calavera | `calavera:45876` | Same |

## Configuration

### `.env` — same on every host

Copy [`services/snoot/.env.example`](../../services/snoot/.env.example).

| Variable | Purpose |
|----------|---------|
| `BESZEL_KEY` | SSH public key string generated by the **hub** when adding a system. **Same value on every host** — the key belongs to the hub, not each agent. Find it in the `KEY=` field of the `docker run` snippet shown by the Beszel UI's "Add System" dialog. |
| `BESZEL_TOKEN` | Per-system token also surfaced by the hub when adding a system. |

Both values are blank in the `.env.example` and must be filled in **after** the hub is running.

### Image source

Pull from Docker Hub: `henrygd/beszel-agent`. Don't switch to `ghcr.io/henrygd/beszel-agent` — that mirror returns `denied` for unauthenticated pulls even when the package is public. Same constraint applies to the hub image.

## Operations

```bash
loft-ctl start snoot
loft-ctl rebuild snoot
sudo docker logs snoot --tail 30
```

There's no `loft-ctl health snoot` URL — snoot has no web UI. Health is verified from the hub side: in `https://beszel.loft.hsimah.com`, each system should show "Online" with fresh data.

### Provisioning a new host

1. On [space-needle](../hosts/space-needle.md), open Beszel → **Add System** for the new host. Copy the `KEY=` value.
2. On the new host, after cloning the repo:

   ```bash
   sudo cp services/snoot/.env.example services/snoot/.env
   sudo sed -i "s|^BESZEL_KEY=.*|BESZEL_KEY=<value-from-hub>|" services/snoot/.env
   ```

3. `sudo bash setup.sh` (or `loft-ctl start snoot`).
4. Back on the hub, set the system's connection — bare hostname:45876 for remote hosts, `host.docker.internal:45876` for space-needle's local agent.

The same `BESZEL_KEY` works for every host — copy the same value into each `.env`. The `BESZEL_TOKEN` may be system-specific (the UI labels it as such when generated).

## Debug & Troubleshooting

### Agent not reporting — hub shows the system as Offline

**Checks (from the hub host, space-needle):**

```bash
# Local agent reachable?
nc -zv host.docker.internal 45876   # from space-needle host
# Or, from inside the hub container:
sudo docker exec beszel nc -zv host.docker.internal 45876

# Remote agent reachable?
nc -zv viking 45876
nc -zv fjord 45876
nc -zv calavera 45876
```

**Checks (on the host running the silent agent):**

```bash
sudo docker logs snoot --tail 50

# Is the port actually open?
sudo ss -tlnp | grep 45876
# Should show snoot listening on 0.0.0.0:45876
```

Common causes:
- The host firewall blocks 45876 (rare on the loft hosts — none run ufw by default; check `sudo iptables -L INPUT` if in doubt).
- `BESZEL_KEY` mismatch — the value in `.env` differs from what the hub thinks the system's key should be. Regenerate from the hub UI and resync.
- The hub system entry has the wrong host:port — e.g. `localhost` for space-needle, which won't work from inside the bridge.

### `space-needle` is the only host that's offline

Symptom: fjord/viking/calavera are happy in the hub, space-needle is offline.

**Cause:** The space-needle system in the hub is configured with `localhost:45876` or `127.0.0.1:45876`. From inside the bridge-networked hub container, that's the container's own loopback, not the host network.

**Fix:** Edit the system in Beszel, set host = `host.docker.internal`, port = `45876`. The `extra_hosts: host.docker.internal:host-gateway` block on `beszel` in [`services/houstn/docker-compose.yml`](../../services/houstn/docker-compose.yml) is what makes this resolve to the host gateway.

### `denied` on `docker pull`

**Cause:** Image was specified as `ghcr.io/henrygd/beszel-agent` instead of Docker Hub.

**Fix:** Edit `services/snoot/docker-compose.yml` so `image:` is `henrygd/beszel-agent`. `loft-ctl rebuild snoot`.

### Per-container Docker stats missing

The agent exposes Docker stats only when it can read `/var/run/docker.sock`. If you see host CPU/RAM but no containers in the Beszel dashboard:

```bash
# Confirm the bind mount is present
sudo docker inspect snoot --format '{{range .Mounts}}{{.Source}} → {{.Destination}}{{println}}{{end}}'
# Expect: /var/run/docker.sock → /var/run/docker.sock

# Confirm the agent can actually read the socket
sudo docker exec snoot ls -l /var/run/docker.sock
```

If the socket isn't mounted, the compose was edited. Restore the volume and rebuild.

### Agent reports the wrong hostname

Beszel labels metrics by whatever the agent reports its hostname as — usually `/etc/hostname`. After a `hostnamectl set-hostname` change, restart snoot so it picks up the new value:

```bash
loft-ctl rebuild snoot
```
